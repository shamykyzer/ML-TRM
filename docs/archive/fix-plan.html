02Fix A — OOM on maze (gpu_config override)

Confirmed variant (a) chosen 2-line change

Root cause

main.py:56 calls apply_gpu_overrides(config) for every run. Inside src/utils/gpu_config.py:104–116:

def apply_gpu_overrides(config: ExperimentConfig) -> None:
    gpu_profile = detect_gpu()
    task = config.data.dataset if hasattr(config.data, "dataset") else "sudoku"
    task_profile = gpu_profile.get(task, gpu_profile.get("sudoku", {}))
    if "batch_size" in task_profile:
        config.training.batch_size = task_profile["batch_size"]  # unconditional overwrite

The RTX 5070 profile's "maze": {"batch_size": 16} was tuned for TRM (~7M params). Your Qwen-maze YAML explicitly sets batch_size: 2, grad_accum_steps: 8 — but the override silently clobbers batch_size=2 to batch_size=16.

YAML intent

batch_size: 2          # per-step fits 11.9 GB
grad_accum_steps: 8   # effective batch 16

Actual (after override)

batch_size: 16         # 8× VRAM, lm_head 4.3 GB alone
grad_accum_steps: 8   # effective batch 128 — unintended

Variant chosen: (a) gate override on model_type

Variant Change Pros Cons
(a) Early-return from apply_gpu_overrides when model_type ∈ {LLM_FINETUNE, LLM_DISTILL} 2-line change; restores function to designed scope; YAML stays authoritative for LLMs None meaningful
(b) Add llm_sudoku/llm_maze keys to every GPU profile Centralises batch tuning Second source of truth alongside YAML → drift; higher maintenance
(c) Honor YAML when "explicitly set" via sentinel Most general Invasive schema change; needs to propagate through config loader
Why (a): The design doc explicitly lists apply_gpu_overrides under "Kept Infrastructure" for trainer_official.py. It was never positioned as a universal auto-tuner — main.py just calls it for every code path out of historical convenience. Gating restores the function to its designed scope without touching TRM behaviour.
★ Insight
  • Each LLM YAML has hand-tuned batch_size × grad_accum_steps pairs specific to that model's parameter count and sequence length. Llama uses 8 × 2, Qwen-maze uses 2 × 8, GPT-2 uses 32 × 1. A shared GPU table can't capture this — the tradeoffs are model-specific, not GPU-specific.
  • The running trm_official_sudoku training uses TRM_OFFICIAL_SUDOKU model type — variant (a) does not touch its code path, so editing gpu_config.py is safe while that run is live.

03Fix B — eval off-by-one + missing cell_accuracy

Expanded scope Blocks thesis ~10-line change

Root cause — causal-LM shift mismatch

HuggingFace Qwen2ForCausalLM.forward(input_ids, labels=…) shifts internally during loss:

shift_logits = logits[..., :-1, :]   # positions 0..T-2
shift_labels = labels[..., 1:]       # positions 1..T-1
loss = CrossEntropyLoss()(shift_logits, shift_labels)

So training teaches: logits[i] predicts labels[i+1].

But eval in trainer_llm.py:181–195 compares without shifting:

outputs = self.model(input_ids=inputs)
preds = outputs.logits.argmax(-1)            # preds[i] = argmax logits[i]
mask = labels != 0
puzzle_correct = ((preds == labels) | ~mask).all(dim=-1)
# ^ compares preds[i] to labels[i] — off-by-one vs training

Consequence: val/puzzle_acc is stuck at 0 regardless of whether the model learned anything. The loss curve (2.31 → 1.95) shows training is working; the accuracy metric is just measuring the wrong thing.

Expanded scope (based on plan.md)

Original plan: fix the shift. plan.md review added two more items:

  • 1. Shift preds and labels by one position so eval matches training.
  • 2. Add cell_accuracy (per-token accuracy on positions where label ≠ 0). trainer_trm.py already reports this; the LLM trainer was missing it for no principled reason.
  • 3. Log both to W&B under val/puzzle_acc and val/cell_acc — matches the panel structure plan.md §A Task 6 established.

Proposed eval loop (after fix)

@torch.no_grad()
def evaluate(self) -> dict:
    self.model.eval()
    total_puzzles_correct = 0
    total_cells_correct = 0
    total_cells_graded = 0
    total_puzzles = 0

    for inputs, labels in self.val_loader:
        inputs = inputs.to(self.device)
        labels = labels.to(self.device)

        outputs = self.model(input_ids=inputs)
        preds = outputs.logits[:, :-1, :].argmax(-1)   # shift preds
        labels_shifted = labels[:, 1:]                    # shift labels
        mask = labels_shifted != 0                         # True where graded

        puzzle_correct = ((preds == labels_shifted) | ~mask).all(dim=-1)
        total_puzzles_correct += puzzle_correct.sum().item()
        total_cells_correct   += ((preds == labels_shifted) & mask).sum().item()
        total_cells_graded    += mask.sum().item()
        total_puzzles         += inputs.shape[0]

    return {
        "puzzle_acc": total_puzzles_correct / max(1, total_puzzles),
        "cell_acc":   total_cells_correct   / max(1, total_cells_graded),
    }
Why cell_accuracy matters for the rubric: the Experimental Evaluation band (30%, 70-100%) requires "fully justified methodology, excellent appraisal and evaluation." Puzzle_acc stuck near zero across all LLMs is a single uninformative comparison. Cell_acc shows how the models fail — e.g., "GPT-2 reaches 22% per-cell accuracy (2× random baseline of 11%, one-of-9-digits) but can't string 40 correct digits together to solve a puzzle." That's real appraisal. The report needs it.
★ Insight
  • Loss 1.95 nats/token → perplexity ≈ 7. Over 11 valid classes, that means the model concentrates ~1⁄7 probability mass on the correct digit. Translated through argmax, cell_acc should land in the 20-40% range after Fix B.
  • Puzzle accuracy scales brutally: if cell_acc on blanks is 30%, puzzle_acc = 0.3040 ≈ 10-21. So even the fixed metric will report essentially zero puzzle_acc — which is exactly the thesis claim, now defensibly measured rather than bug-induced.
  • The existing llm_qwen2_5_0_5b_sudoku_latest.pt checkpoint was trained against the correct shifted loss — weights are good. Only the evaluation was broken. Re-evaluating that checkpoint with the fix requires no retraining.
