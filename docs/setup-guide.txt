ML-TRM Setup Guide
===================
Full setup from scratch for training TRM-Sudoku on Windows.

1. PREREQUISITES
----------------
- Python 3.11+
- NVIDIA GPU with CUDA support
- Git

2. BOOTSTRAP EVERYTHING WITH start.py
--------------------------------------
    python start.py

start.py is a stage-aware onboarding script. Run it once, it handles the next missing
setup stage (venv -> .env -> wandb -> data), tells you what it did, and exits. Re-run
until it prints the training menu.

Stages:
    1. venv   - creates ~/.venvs/ml-trm + installs torch (cu128) + requirements.txt
    2. env    - copies .env.example to .env (you then edit it with your WANDB_API_KEY)
    3. wandb  - prompts you to run `wandb login`
    4. data   - downloads Sudoku + Maze datasets from HuggingFace

Check status anytime:
    python start.py status

Skip wandb and continue without tracking:
    python start.py --skip-wandb

Note: RTX 50-series (5070, 5080, 5090) need CUDA 12.8 (cu128). Older CUDA versions
(cu124) will fail with "no kernel image is available for execution on the device"
because Blackwell uses compute capability sm_120. start.py installs cu128 by default.

3. FILL IN .env
----------------
After stage 2 copies .env.example to .env, edit .env with at minimum:

    WANDB_API_KEY=<your key from https://wandb.ai/authorize>

Optional overrides (blank = disabled):
    TRM_WANDB_ENTITY=<username or team>
    TRM_HF_REPO_ID=<you>/ml-trm-checkpoints     # for HF Hub checkpoint sync
    TRM_ROLLING_CHECKPOINT_DIR=D:/ml-trm-backups # external drive backups

The .env loader is wired into src/utils/config.py - explicit YAML values always win
over .env, so machine-specific fields default to "" in the committed YAMLs.

4. TRAIN
--------
Activate the venv first:

    # Windows
    "%USERPROFILE%\.venvs\ml-trm\Scripts\activate"
    # Linux/Mac
    source .venv/bin/activate

Then run:

    python main.py --mode train --config configs/trm_official_sudoku.yaml

Config: configs/trm_official_sudoku.yaml
- N_sup=16 (paper default, "Less is More" Jolicoeur-Martineau 2025)
- batch_size is auto-tuned per GPU via src/utils/gpu_config.py
- 60K epochs with ACT (Adaptive Computation Time) learning to stop early

To resume from checkpoint:
    python main.py --mode train --config configs/trm_official_sudoku.yaml \
        --resume models/sudoku-att/latest.pt

6. HUGGINGFACE HUB CHECKPOINT SYNC
------------------------------------
Checkpoints are too large for Git. They sync to HuggingFace Hub instead.

One-time login:
    & "$env:USERPROFILE\.venvs\ml-trm\Scripts\hf.exe" auth login

Checkpoints auto-upload during training when hf_repo_id is set. The committed YAMLs
leave it blank; set it via .env instead:

    TRM_HF_REPO_ID=<you>/ml-trm-checkpoints

The repo is created as PRIVATE automatically on first training run.

Uploads happen at:
- Every save_interval epochs (periodic checkpoint)
- When a new best accuracy is reached (best.pt)
- At the end of training (latest.pt)

To download checkpoints on another machine:
    hf download shamykyzer/ml-trm-checkpoints --local-dir models/sudoku

8. HOW HUGGINGFACE HUB CHECKPOINT SYNC WORKS
----------------------------------------------
Why use it?
- Git is NOT suitable for large binary files (checkpoints are 50-100MB+). Pushing them
  to GitHub bloats the repo history permanently and slows down cloning.
- HuggingFace Hub is designed for ML model files. It uses LFS-like storage, supports
  versioning, and handles large files efficiently.
- Checkpoints upload automatically during training with no extra commands needed.

How it works in the code:
- trainer_trm.py imports HfApi from huggingface_hub
- On trainer init, if hf_repo_id is set in config, it creates a private HF repo
- Every time _save_checkpoint() runs, it uploads the .pt file to HF Hub right after
  saving locally. Each upload is a separate commit, so you get full version history.
- If the upload fails (e.g. no internet), training continues — it just prints a warning.

What gets uploaded and when:
- Every 25 epochs: epoch_25.pt, epoch_50.pt, etc. (periodic snapshots)
- On new best accuracy: best.pt (overwrites previous best)
- End of training: latest.pt (final state)

How to use across machines:
- Machine A (training): set TRM_HF_REPO_ID in .env and train. Uploads happen auto.
- Machine B (resume): download checkpoints, then resume training:
      hf download <you>/ml-trm-checkpoints --local-dir models/sudoku-att
      python main.py --mode train --config configs/trm_official_sudoku.yaml \
          --resume models/sudoku-att/latest.pt

How to disable:
- Leave TRM_HF_REPO_ID blank in .env (or omit it entirely). Training works exactly
  the same, just without uploading.

Comparison with alternatives:
- Git LFS: requires setup, charges for bandwidth, still bloats clone size
- OneDrive/Google Drive: syncs everything including temp files, no version control
- HF Hub: purpose-built for ML, free private repos, versioned, easy CLI download

9. USEFUL COMMANDS
-------------------
    python start.py                 # bootstrap / continue onboarding
    python start.py status          # show which setup stages are ready

    # Train (activate venv first)
    python main.py --mode train --config configs/trm_official_sudoku.yaml
    python main.py --mode train --config configs/trm_official_maze.yaml
    python main.py --mode train --config configs/llm_config.yaml     # GPT-2
    python main.py --mode train --config configs/llm_qwen.yaml       # Qwen2.5-0.5B
    python main.py --mode train --config configs/llm_smollm.yaml     # SmolLM2-360M
    python main.py --mode train --config configs/llm_llama.yaml      # Llama-3.2-1B

    # Evaluate
    python main.py --mode eval --config configs/trm_official_sudoku.yaml \
        --checkpoint models/sudoku-att/best.pt

    # Resume
    python main.py --mode train --config configs/trm_official_sudoku.yaml \
        --resume models/sudoku-att/latest.pt
